Лучший проект на микроконтроллере в компании БСВТ новые-технологии:

Обработка данных от визиров/прицелов, расчёт баллистики, наведение, интеграция CAN/SPI в системы управления огнём):

16 дискретных цифровых входов (DI) — для приёма сигналов от концевых выключателей, датчиков положения, систем безопасности, триггеров и интерфейсов с legacy-оборудованием (с аппаратной защитой от помех и фильтрацией дребезга).
16 аналоговых входов (AI) — для приёма сигналов от аналоговых датчиков (угловые положения, дальномеры, акселерометры, гироскопы, датчики температуры/давления и т.д.), с комбинацией встроенных 12-битных ADC-модулей (до 2 MSPS) и внешних высокоточных АЦП (например, ADS1256 / ADS1115 по SPI/I²C) для достижения требуемого количества каналов и повышения разрешения/стабильности.
Внешний ЦАП (по SPI, например, 12–16 бит MCP492x / DAC856x) — для формирования аналоговых управляющих сигналов (коррекция прицела, приводы наведения, имитация выходов на legacy-системы).
CAN-шина (встроенный контроллер CAN 2.0B с внешним трансивером) — для надёжного обмена данными в реальном времени с датчиками, исполнительными механизмами, другими блоками БРЭО (бортовое радиоэлектронное оборудование) и внешними системами (в том числе в шумной промышленной/боевой среде).
Несколько SPI-интерфейсов — для высокоскоростного межмодульного обмена: связь с расширителями GPIO, внешними АЦП/ЦАП, флеш-памятью, дисплеями индикации, модулями обработки видео/изображений.

CAN
Использовал встроенный CAN-контроллер (CAN 2.0 A/B, до 1 Мбит/с, поддержка extended identifiers). Подключал внешний трансивер (MCP2551 / MCP2562 или аналог) с гальванической развязкой и защитой от переходных процессов (TVS-диоды, common-mode choke).
Передавал:
телеметрию от датчиков положения башни и стабилизатора (углы азимута/возвышения, угловые скорости),
статусы исполнительных механизмов (приводов ГН и ВН),
команды наведения и коррекции.
Реализовал:
аппаратные и программные фильтры сообщений (message objects с масками),
обработку ошибок (bus-off automatic recovery, error counters),
приоритетную арбитражную схему (критические команды наведения имеют более высокий приоритет, чем диагностические сообщения).

Ethernet
Использовал встроенный Ethernet MAC + PHY (RMII-интерфейс) микроконтроллера TM4C129x.
Применял лёгкий TCP/IP-стек lwIP (адаптированная версия от TI или vanilla с минимальными модификациями).
Работал преимущественно по UDP-протоколу (connectionless, низкий overhead, подходит для реального времени):
передача больших объёмов данных: сырые кадры от оптико-электронного визира/прицела, высокоточная телеметрия (позиция, ориентация, поправки), короткие видео-сниппеты или JPEG-кадры,
MAVLink over UDP (стандартный транспорт для связи onboard → ground control station).
Поддерживал также TCP для надёжных сессий (например, передача параметров конфигурации или логов).
Обеспечивал: QoS-подобное поведение (приоритет пакетов с командами), CRC на уровне приложения, retransmission при потере критических сообщений.

MAVLink
Применял протокол MAVLink (v1.0 / v2.0) как унифицированный слой поверх разных физических транспортов:
UART — для локальной отладки и сервисного доступа,
CAN — для внутренних датчиков и legacy-устройств (через custom-мост MAVLink → CAN),
Ethernet / UDP — для внешней высокоскоростной связи с GCS (QGroundControl, Mission Planner) или другими onboard-системами.
Реализовал парсинг и формирование ключевых сообщений:
HEARTBEAT (системный статус, тип аппарата, режим работы),
ATTITUDE / LOCAL_POSITION_NED / GLOBAL_POSITION_INT (данные для стабилизации и наведения),
COMMAND_LONG / COMMAND_INT (управление приводами, установка параметров),
MISSION_* (если использовались задачи типа «навести на координаты цели» или waypoint-наведение),
PARAM_* (чтение/запись калибровок, настроек СУО).
Разработал bidirectional-мосты:
MAVLink → CAN (для интеграции с устаревшими блоками по CAN),
MAVLink over UDP/Ethernet (для связи с внешними системами и GCS).
Использовал: официальную C-библиотеку mavlink/c-library, generated код из XML-диалектов (common.xml, ardupilotmega.xml), собственные расширения для военных/специфических сообщений.

###### Жизненный цикл лицензии ######

Пользователь впервые запускает программу
→ вводит email + пароль
→ приложение отправляет их на /api/joy/login
Сервер отвечает:Вариант А — пользователь новый → создаёт триал на 60 минут в сутки
Вариант Б — пользователь уже есть → проверяет пароль
→ проверяет, не занято ли это устройство (HWID) другим аккаунтом
→ если всё ок → добавляет HWID в список разрешённых (максимум 3 за 7 дней)
→ выдаёт токен (это как временный пропуск)
Приложение сохраняет токен зашифрованно в файл login.dat
Каждый раз при запуске (и периодически во время работы) приложение делает:
POST /api/joy/validate
(передаёт токен + deviceId)
Сервер отвечает примерно так:
JSON{
"valid": true,
"type": "trial", // или "monthly", "yearly" и т.д.
"trialRemaining": 43,
"trialLimit": 60,
"message": "Trial mode active"
}
Если токен недействителен / просрочен / HWID не разрешён → сервер говорит «нет» → приложение переходит в оффлайн-режим по последнему известному состоянию.
Когда пользователь пользуется программой → приложение каждые N минут отправляет heartbeat
POST /api/joy/heartbeat
(токен + deviceId + minutesUsed = обычно 1)

Сервер:
проверяет, что токен валидный
проверяет, что это устройство (HWID) действительно привязано к этой лицензии
если триал → списывает минуты за сегодня
если превышен лимит на день → возвращает ошибку
обновляет список IP-адресов (для отладки / аналитики)
иногда возвращает новый токен, если тип лицензии поменялся

Оффлайн-режим (когда нет интернета или сервер не отвечает)
Приложение хранит в файле licstate.dat последнее известное состояние:
это был триал → сколько минут оставалось
это была платная подписка → до какого числа действовала
Пока это состояние не истекло по времени — приложение продолжает работать.
Пример сообщений в оффлайне:
«Trial: 18 min (offline mode)»
«Subscription active - remaining 14d 3h 41m (offline mode)»
«Trial expired (offline mode)» → блокировка


Ключевые защиты (что сложно обойти)
HWID привязано к лицензии → нельзя залогиниться с одного устройства под двумя аккаунтами
Максимум 3 устройства за 7 дней (старые вытесняются только если им > 7 дней)
Триал — только 60 минут в сутки (сбрасывается в 00:00 по серверному времени)
Токен проверяется на сервере → подделать сложно (JWT + секретный ключ)

###### Админ-панель JoyControl (ardu.live/admin) ######
— закрытая зона только для владельца/администраторов:
Что видит и может делать админ:

Главная таблица: все пользователи + их лицензии (тип, статус, дата окончания, остаток триала, IP-адреса, заметки)
Быстрые действия по пользователю:
Активировать 1 год / 2 года / навсегда
Установить кастомную дату окончания
Сбросить на триал (с сохранением HWID)
Обнулить использованные минуты триала сегодня
Вручную изменить остаток минут триала

Управление оптовиками (/admin/joylic):
Список оптовиков
По каждому — таблица всех его кодов (статус: не активировано / активировано кем+когда / заморожено)
Массово: заморозить/разморозить/удалить все коды оптовика
По одному: заморозить/разморозить/удалить код

Создание оптовых кодов (/admin/joyadd):
Выбрать количество (до 100)
Типы: 1 год / 2 года / пожизненно (можно несколько сразу)
Кому: существующий оптовик или новый (ФИО + email)
После создания — список кодов + копирование (все / с типами)


Админ видит, кто где активировал коды, может замораживать остатки, выдавать подписки вручную и полностью контролировать систему лицензий.


###### Обычный пользователь ######

видит в профиле и связанных страницах:
Свои данные: ID, имя, email
Кнопки: Редактировать профиль / Выйти
Ссылки: страховка → управление лицензиями → активировать ключ
Блок личной лицензии: тип (триал/год/2 года/навсегда), статус, дата окончания, остаток минут триала
Если лицензии нет — предложение активировать код
Свои заявки на страховку (с фото чека, статусом «На рассмотрении» / «Одобрено», можно менять фото и удалять до одобрения)

###### Оптовик ######

видит всё то же + дополнительно:
На странице управления лицензиями — блок «Мои оптовые коды»
Список всех своих купленных кодов: код, тип подписки, статус (не активировано / активировано кем+когда / заморожено)
Кнопка «Скопировать все коды» и копирование по одному
Оптовик не может редактировать/замораживать/удалять свои коды — это только админ.
